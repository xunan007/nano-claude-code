# 权限系统

## 为什么需要权限系统

s07 的目标是让每一次工具调用都先经过权限管线，再决定是否真正执行。

模型生成的是执行意图，但意图不能直接等于执行。尤其是 bash、write_file、edit_file、task 这类会改变环境或进一步派发工作的工具，需要一个统一入口做拦截。

**注意：权限清单不会交给模型记忆，这是工程上处理的事情。**

## 权限模式

当前实现保留三个教学模式：

- default：读取类工具按规则自动允许，其他未匹配工具询问用户
- plan：阻止写入类工具，只允许读取和上下文类工具
- auto：读取类工具自动允许，写入类工具仍会询问用户

权限从高到底依次是：auto > default > plan

启动时会先选择模式：

```txt
Permission modes: default, plan, auto
Mode (default):
```

运行中可以用 /mode 切换：

```txt
/mode plan
/mode default
/mode auto
```

## 权限管线

每个 tool call 的执行顺序是：

1. bash 安全校验
2. deny rules
3. mode check
4. allow rules
5. ask user

这对应一个核心原则：安全不是一个 boolean，而是一条 pipeline。

## Bash 安全校验

BashSecurityValidator 会先检查一些高风险模式：

- shell 元字符：;、&、|、反引号、$
- sudo 提权
- 递归删除
- 命令替换
- IFS 注入

其中 sudo 和递归删除会直接 deny。其他模式会升级为 ask，让用户决定是否批准。

## 规则系统

规则格式：

```ts
type PermissionRule = {
    tool: string;
    path?: string;
    content?: string;
    behavior: "allow" | "deny" | "ask";
};
```

默认规则包括：

- 禁止 rm -rf /
- 禁止 sudo \*
- 允许读取文件
- 允许加载 skill

可以用 /rules 查看当前规则。如果用户在权限询问里输入 always，系统会为该工具追加一条临时 allow 规则。

## 接入位置

权限系统接在 ToolRuntime.executeToolCalls 里。

模型返回 tool call 后，runtime 先调用 PermissionManager.check。只有结果是 allow，或者 ask 后用户批准，才会执行对应 handler。

父代理和子代理共享同一个 PermissionManager，所以子代理通过 task 发起的工具调用也会进入同一条权限管线。

## 降权限

ToolRuntime.executeToolCalls 中会对拒绝次数进行判断，如果出现多次拒绝说明模型一直在尝试做危险的动作，这个时候可以考虑切换到 plan 模式，降低权限。
