# 记忆系统

## 为什么需要记忆

s09 的目标是让 agent 记住跨会话仍然有价值的信息。

上下文压缩解决的是“当前长任务怎么继续做”，记忆系统解决的是“下一次会话还应该知道什么”。这两者边界不同：

- compact 保留当前任务状态
- memory 保留跨会话偏好、反馈、非显然项目事实和外部资源指针

记忆不是聊天记录归档，也不是代码索引。能从当前仓库重新读出来的信息，不应该写入记忆。

## 存储结构

记忆文件放在工作区的 .memory 目录下：

```txt
.memory/
  MEMORY.md
  prefer_tabs.md
  review_style.md
  incident_board.md
```

每条记忆是一个独立 Markdown 文件，使用简单 frontmatter：

```md
---
name: prefer_tabs
description: User prefers tabs for indentation
type: user
---

Use tabs for indentation when editing this project.
```

- name：记忆名称，也是 prompt 里展示的标题
- description：一行摘要，方便快速扫描
- type：记忆类型
- 正文：完整记忆内容

MEMORY.md 是自动重建的索引，不需要手写维护。

## 记忆类型

当前支持四类记忆：

- user：用户偏好，比如代码风格、沟通方式、常用工具
- feedback：用户反复纠正过的行为，比如不要做某类操作
- project：不容易从代码直接推导出的项目约定和决策原因
- reference：外部资源指针，比如文档地址、看板地址、仪表盘地址

这些类型会影响 system prompt 里的分组展示，让模型更容易判断记忆的性质。

## MemoryManager

MemoryManager 负责三件事：

- 启动时加载 .memory/\*.md
- 构建可注入 system prompt 的记忆区块
- 保存新记忆并重建 MEMORY.md

核心方法：

- loadAll()
    - 扫描 .memory 目录
    - 跳过 MEMORY.md
    - 解析每个 Markdown 文件的 frontmatter
- loadMemoryPrompt()
    - 按 user、feedback、project、reference 分组
    - 生成 system prompt 片段
- saveMemory(name, description, type, content)
    - 校验 type
    - 把 name 转成安全文件名
    - 写入单条记忆文件
    - 更新内存态
    - 重建索引

## 工具接入

新增 save_memory 工具：

```ts
{
    name: "save_memory",
    description: "Save a persistent memory that survives across sessions.",
    parameters: {
        type: "object",
        properties: {
            name: { type: "string" },
            description: { type: "string" },
            type: {
                type: "string",
                enum: ["user", "feedback", "project", "reference"],
            },
            content: { type: "string" },
        },
        required: ["name", "description", "type", "content"],
    },
}
```

ToolRuntime 只有在传入 memoryManager 时才注册这个工具。这样记忆系统仍然是一个可组合能力，不会把工具运行时和具体领域状态绑死。

父代理和子代理共享同一个 MemoryManager。子代理如果学到真正应该跨会话保留的信息，也可以保存记忆。

## Prompt 注入

PromptBuilder 每次构建 system prompt 时都会调用：

```ts
memoryManager.loadMemoryPrompt();
```

所以新保存的记忆会在下一轮模型调用里可见。

system prompt 还会注入记忆使用准则：

- 用户明确表达长期偏好时，可以保存
- 用户纠正错误行为时，可以保存
- 项目中存在非显然约定时，可以保存
- 外部资源位置值得下次找回时，可以保存
- 代码结构、临时状态、secret 不保存

这一步很重要。没有准则的记忆系统很容易变成“什么都记”，最后 prompt 变脏，记忆也失去价值。

## CLI 命令

s09 新增 /memories 命令：

```txt
/memories
```

如果当前进程已经加载记忆，会输出：

```txt
  [user] prefer_tabs: User prefers tabs for indentation
```

如果没有记忆，会输出：

```txt
  (no memories)
```

启动时也会提示当前是否加载到了已有记忆。

## DreamConsolidator

Dream 是一个后续整理记忆的教学骨架。它不负责首版记忆能力，但说明了长期记忆需要维护，否则会越来越噪。

当前包含 7 个 gate：

- enabled flag 必须开启
- .memory 目录必须存在且有记忆文件
- plan mode 不执行整理
- 距离上次整理至少 24 小时
- 距离上次扫描至少 10 分钟
- 至少积累 5 次 session
- 没有其他进程持有 lock

通过 gate 后执行 4 个 phase：

```txt
Orient: scan MEMORY.md index for structure and categories
Gather: read individual memory files for full content
Consolidate: merge related memories, remove stale entries
Prune: enforce 200-line limit on MEMORY.md index
```

当前实现只打印 phase，先理解 Dream 的位置和边界。真正的合并、去重、裁剪可以以后再接入模型调用。

## 执行流

```txt
process start
    -> MemoryManager.loadAll()
    -> PromptBuilder.parent/subagent injects memories
    -> model sees user asks to remember something
    -> model calls save_memory
    -> MemoryManager writes .memory/<name>.md
    -> MemoryManager rebuilds .memory/MEMORY.md
    -> next model turn sees updated memory prompt
```

记忆系统的关键点不是“把信息写进文件”，而是判断什么信息值得写进去。
