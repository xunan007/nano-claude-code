# s01-s06 代码重构

## 对比基准

本说明基于当前分支和 `feat/s06` 的对比。

`feat/s06` 已经包含 message 转换、技能系统、todo、子代理、上下文压缩等能力，但这些能力主要堆在 `src/index.ts` 里。当前分支的核心目标不是改业务能力，而是把 s01 到 s06 逐步累积出来的单文件实现拆成更清楚的模块边界。

## 原来的问题

`feat/s06` 里的 `src/index.ts` 同时承担了太多职责：

- 类型定义、配置常量、工具定义、工具执行都在同一个文件里
- message normalize、OpenAI-compatible 转换、DeepSeek reasoning 字段处理和最终文本提取混在一起
- compact 状态通过函数参数在工具调用链路里传来传去，调用关系不直观
- todo、skill、task、compact 都直接散落在全局 handler 里，新增工具时容易漏改工具列表或 handler 列表
- `runOneTurn` 同时处理模型调用、工具执行、todo reminder、compact 触发和消息写回，后续继续扩展会越来越难维护
- 构建仍直接依赖 `tsc` 输出，拆分多模块后需要更明确的 ESM 打包入口

这类问题短期不影响功能跑通，但会让后续继续加权限控制、日志、hook、工具分组或多模型适配时变得很别扭。

## 重构后的模块

当前分支把原本集中在 `src/index.ts` 的逻辑拆成了多个模块：

```txt
src/index.ts             CLI 入口、读取用户输入、组装依赖、写 message trace
src/types.ts             内部消息、模型响应、计划、技能、compact、loop state 类型
src/config.ts            工作目录、token/上下文阈值、输出目录、trace 路径等配置
src/model-client.ts      DeepSeek/OpenAI-compatible 模型调用封装
src/message-codec.ts     内部 Message 和 OpenAI message 之间的转换
src/tool-runtime.ts      工具定义、工具注册、工具执行、文件和 bash 工具实现
src/compact-manager.ts   compact 状态、大输出落盘、微压缩、完整压缩
src/todo-manager.ts      todo 状态、计划校验、计划渲染、reminder 判断
src/skill-registry.ts    扫描 .skills、读取 skill frontmatter、按需加载 skill 正文
src/prompt-builder.ts    父代理和子代理 system prompt 构建
src/agent-loop.ts        父代理 loop、单轮执行、子代理执行流程
```

拆分后的入口变薄了：`index.ts` 只负责 CLI 生命周期和依赖装配，真正的 agent 行为交给 `AgentLoop`。

## 执行链路

重构后的主流程是：

```txt
main()
  -> createAgentLoop()
      -> MessageCodec
      -> ModelClient
      -> CompactManager
      -> SkillRegistry
      -> TodoManager
      -> PromptBuilder
      -> AgentLoop
  -> history.push(user message)
  -> agentLoop.createInitialState(history)
  -> agentLoop.run(state)
      -> runOneTurn()
          -> CompactManager.microCompact()
          -> CompactManager.shouldCompact()
          -> ModelClient.createMessage()
              -> MessageCodec.toOpenAIMessages()
              -> DeepSeek/OpenAI-compatible API
              -> MessageCodec.fromOpenAIMessage()
          -> ToolRuntime.executeToolCalls()
          -> TodoManager reminder
          -> manual compact if compact tool was called
  -> writeMessageTrace(history)
  -> MessageCodec.extractAssistantTexts()
```

子代理也统一由 `AgentLoop.runSubagent()` 编排。父代理和子代理都会使用 `ToolRuntime`，但构造参数不同：

- 父代理 runtime：基础文件工具、`load_skill`、`todo`、`task`、`compact`
- 子代理 runtime：基础文件工具、`load_skill`

这样仍然保持了原来的隔离原则：子代理共享文件系统，但不共享父代理会话历史，也不能继续递归创建子代理或修改父代理 todo。

## 关键边界变化

### MessageCodec

`MessageCodec` 接管所有消息格式转换：

- 内部 `text / tool_use / tool_result` block 转成 OpenAI-compatible message
- OpenAI/DeepSeek assistant message 转回内部 block
- 合并相邻同 role 消息
- 补齐缺失的 tool result
- 保留 DeepSeek 的 `reasoning_content`
- 提取最终 assistant 文本

agent loop 不再关心 API message 的细节，只处理内部 `Message`。

### ToolRuntime

`ToolRuntime` 把工具定义和 handler 注册放在同一个对象里，避免工具 schema 和执行逻辑分散维护。

它根据构造参数决定启用哪些能力：

- 默认启用 `bash/read_file/write_file/edit_file/load_skill`
- 传入 `todoManager` 时启用 `todo`
- 传入 `runSubagent` 时启用 `task`
- `enableCompactTool` 为 true 时启用 `compact`

文件路径安全检查、bash 超时、大输出持久化、读文件 recent file 追踪也都集中在这里。

### CompactManager

`CompactManager` 接管 compact 相关状态和行为：

- `estimateContextSize`
- `shouldCompact`
- `trackRecentFile`
- `persistLargeOutput`
- `microCompact`
- `compactHistory`

大输出仍会写入 `.task_outputs/tool-results`，完整压缩前仍会把 transcript 写入 `.transcripts`。区别是 compact state 不再挂在 `LoopState` 里，而是由 manager 自己维护。

### AgentLoop

`AgentLoop` 现在是流程编排层：

- 自动 compact 前置处理
- 调模型
- 写回 assistant message
- 执行工具
- 追加 tool result
- todo reminder 后处理
- manual compact 后处理
- 子代理 loop

`LoopState` 因此被收窄为 agent loop 自身状态：

```ts
type LoopState = {
    messages: Message[];
    turnCount: number;
    transitionReason?: "tool_result";
};
```

Todo、skill、compact、model client 这些服务依赖不再塞进 state，而是由 `AgentLoopOptions` 注入。

## 构建变化

构建链路也跟着调整：

- `package.json` 的 `build` 从 `tsc -p tsconfig.build.json` 改为 `tsup src/index.ts --format esm --dts --sourcemap --clean`
- 新增 `tsup.config.ts`
- `tsconfig.json` 从 `NodeNext` 调整为 `ESNext` + `Bundler`
- `tsconfig.build.json` 设置 `noEmit: true`，类型检查和实际打包职责分开
- 新增 `tsup` 开发依赖

也就是说，`tsc` 主要负责类型检查，实际产物由 `tsup` 从 `src/index.ts` 入口打包。

## 保持不变的行为

这次重构尽量保持 s06 的外部行为不变：

- CLI 入口仍是 `src/index.ts`
- prompt 仍显示 `s06 >>`
- `.env` 仍在启动时加载
- `DEEPSEEK_API_KEY` 仍是必需环境变量
- 默认模型仍是 `deepseek-v4-flash`
- 内部消息格式仍使用 `text / tool_use / tool_result`
- `load_skill` 仍按 `.skills/**/SKILL.md` 工作
- `todo` 仍限制最多 12 项，并要求最多一个 `in_progress`
- `task` 仍创建干净上下文的子代理
- 自动 compact 和手动 compact 仍保留
- message trace 仍写入 `.debug/messages-*.json`

## 后续问题

当前重构解决了单文件过大的问题，但还有几个可以继续推进的点：

- `AgentLoop.runOneTurn()` 仍然直接写死了 compact、todo reminder、manual compact 等前后处理，后续可以抽成 hook
- `ToolRuntime` 目前既是工具 adapter，又直接持有文件工具、skill、todo、task、compact 的组装逻辑，后续可以进一步拆分 domain service 和 tool adapter
- `ModelClient` 每次请求都会 new 一个 OpenAI client，可以考虑在构造时初始化客户端
- 工具执行目前仍是串行，之前留下的并发安全分类没有继续落地
- 文档和测试可以按模块补齐，尤其是 `MessageCodec`、`ToolRuntime`、`CompactManager` 这几个容易出现边界问题的模块

## 总结

这次重构的本质是把 s01-s06 的能力从“能跑的单文件实验代码”整理成“可以继续演进的小型 agent runtime”。

功能层面没有刻意改变 s06 的使用方式；结构层面把模型适配、消息转换、工具运行、上下文压缩、技能加载、todo 管理和 loop 编排拆开，为后续继续做 hook、权限、测试、多模型和更复杂工具系统留出空间。
