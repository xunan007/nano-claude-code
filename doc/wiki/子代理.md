# 子代理

## 为什么需要子代理

s04 的目标是给父代理的上下文减负。父代理只负责判断是否需要派发任务，子代理在一个全新的消息数组里完成探索、读写文件或小型修改，最后只把摘要交还给父代理。

关键点是：子代理共享文件系统，但不共享父代理的会话历史。

## 执行时机

当父 Agent 认为有必要拉起子 agent 的时候会调用 task 工具。

```ts
Parent messages=[...]      Subagent messages=[]
        |                         |
        | task(prompt)            |
        |------------------------>|
        |                         | call bash/read/write/edit
        | summary                 |
        |<------------------------|
```

## 工具隔离

- 父代理工具：`bash`、`read_file`、`write_file`、`edit_file`、`todo`、`task`
- 子代理工具：`bash`、`read_file`、`write_file`、`edit_file`
- 子代理没有 `task`，避免递归创建子代理
- 子代理没有 `todo`，避免污染父代理当前会话计划

## 实现入口

- `runSubagent(prompt)`
    - 创建 `subMessages = [{ role: "user", content: prompt }]`
    - 使用 `SUBAGENT_SYSTEM` 和 `CHILD_TOOLS` 调用模型
    - 子代理工具结果继续写回 `subMessages`
    - 循环结束后提取最后一次模型回复中的文本摘要
- `task` handler
    - 从参数里读取 `prompt` 和 `description`
    - 调用 `runSubagent(prompt)`
    - 将摘要作为 `tool_result` 写回父代理消息

父代理不会看到子代理中间的工具调用历史，因此主上下文更清爽。
