# 后台任务

## 核心理念

没有这种后台任务的调度策略，原来的执行是线性的：

```
模型 -> 调工具 -> 等工具执行完 -> 拿结果 -> 模型继续
```

有了后台任务后，变成：

```
模型 -> background_run -> 立刻拿到 background id
后台命令继续跑
模型可以继续处理别的事
后台命令完成后 -> 结果进入通知队列
下一次模型调用前 -> AgentLoop 把结果写回上下文
模型从上下文里的 <background-results> 看到结果
```

**所以核心就是模型看到的是一整段上下文，不是只能像程序一样按线性调用栈执行。只要信息还在上下文里，而且有清晰标识，模型就可以把前后分散的信息关联起来。**

不过这里的实现过于简单，如果后续的任务对于当前工具的结果强依赖，那么这里还要实现任务调度，比如阻塞调用之类。

可见，Harness Engineering 越来越像是给 LLM 用的 OS。
